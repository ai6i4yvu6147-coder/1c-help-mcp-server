## Agent onboarding (контекст для ИИ)

### Коротко о проекте

Проект индексирует распакованную справку платформы 1С (`shcntx_ru`, `shlang_ru`, `shquery_ru`) в SQLite и предоставляет MCP-инструменты для синтакс-помощника BSL, языка запросов и эвристической проверки кода.

### Ключевые политики проекта (не нарушать)

- **NO_DB_MIGRATIONS**: никогда не писать миграции/конвертации существующих SQLite-баз в `databases/`. После изменений схемы/логики импорта базы **всегда пересоздаются** через `admin_tool` из исходников справки.
- **Тестирование**: функциональная проверка делается на «рабочем» MCP после пересборки сервера пользователем и переподключения MCP в агенте. Проверка — **только через вызов MCP tools**, начиная с `list_help_versions`; без прямого чтения SQLite и без Python-скриптов-обходных путей.
- **Исходники vs runtime (portable)**: исходники в репозитории не должны содержать runtime-состояние (`databases/*.db`). Базы живут в portable-экземпляре (`../1c_help_mcp_server_Portable/databases/`) и создаются через Admin.
- **Распределение ответственности**: агент меняет исходники; пользователь пересобирает portable/server и при необходимости пересоздаёт базы; агент проверяет результат через MCP tools.
- **Парсер: опираемся на реальные HBK**: при доработках `shared/help_parser.py` / `shared/query_parser.py` нужно смотреть реальные HTML из распакованной справки. Источники лежат вне репозитория (папка с `shcntx_ru` / `shlang_ru` / `shquery_ru`).
- **BSL vs язык запросов**: для встроенного языка — `get_syntax`, `search_syntax`; для текста запросов (`ВЫБРАТЬ`, `ЕСТЬNULL`) — `get_query_syntax`, `search_query`, `list_query_topics`.

### Термины

- **«Реальный MCP»**: сервер, **уже подключён в IDE**, проверка — фактические вызовы его tools.
- Имя подключения в IDE может не совпадать с названием exe. Ориентируйтесь на наличие tools (`list_help_versions`, `get_syntax`, …) и их ответы.
