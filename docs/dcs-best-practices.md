# Best-practices СКД → что учесть в конструкторе

Смысловой слой: **зачем** нужны настройки СКД и как их проектировать оптимально. Источник —
пользовательская выдержка (из общих статей), **критически размечена**: часть тезисов верна,
часть упрощена. Цель — не закодировать чужие ошибки, а вытащить верное ядро и привязать к
нашему инструментарию (`build_dcs_*`, `describe`, `set_dcs`).

Разметка привязки: ✓ уже есть · ○ в дизайне · △ будущий билдер/валидатор · ✎ авторинг
запроса (текст пишет агент, не билдер).

## Ядро: параметры vs отборы

Самый ценный тезис выдержки, и он в целом верен, но с уточнением.

- **Параметры (`&Параметр`)** — жёсткое условие *на уровне запроса*. Критично для
  **виртуальных таблиц** остатков/оборотов: `Период`/`ДатаНачала`/`ДатаОкончания` обязаны
  быть параметрами ВТ. Передать период «отбором» — СУБД посчитает итоги на текущую дату, а
  потом отфильтрует на сервере 1С → большие тормоза. **[верно, важно]**
- **Отборы (КомпоновщикНастроек)** — фильтр поверх результата; СКД старается протолкнуть
  его в SQL (`ГДЕ`/параметры ВТ). Для **базовых реквизитов** (Организация, Контрагент,
  Склад, Номенклатура) — стандарт 1С: не писать `ГДЕ Организация = &Организация`, а
  использовать механизм отборов; СКД подставит его в SQL эффективно. **[верно]**

**Критика формулировок выдержки [осторожно]:** таблица «параметр = СУБД сразу отсекает /
отбор = СКД пытается внедрить (может тормозить на сервере)» — упрощение. Корректно
построенный отбор по индексированному базовому реквизиту так же проталкивается в `ГДЕ`
SQL. Настоящий водораздел не «параметр быстрый / отбор медленный», а:
- что **меняет саму выборку ВТ** (период, валюта учёта, ветвление логики) → **параметр**;
- что даёт **вариативность условия пользователю** (Равно/Содержит/В списке/Заполнено) по
  обычному полю → **отбор**.
Это перформанс-эвристика, зависит от СУБД/версии → **guidance, не жёсткая проверка**
(запрещающей валидации не делаем — созвучно «отбор это шаблон»).

**Привязка:**
- ✓ `describe`: guidance раздела `parameter` — «период/дата ВТ и ветвящая логика →
  параметр; базовые реквизиты → отбор, не зашивать в `ГДЕ`» — реализовано в
  `vocabulary.py` (`describe(unit='dcs', name='parameter')`).
- ○ `set_dcs` дефолт: базовый реквизит как отбор + вынос в польз. настройки; период — как
  параметр (канонический период-трио уже есть).

## Авторинг запроса

- **`{}` необязательные условия** — `{ГДЕ Контрагент = &Контрагент}`: если пользователь не
  заполнил, СКД вырежет условие из SQL. **[верно]** → ✎ агент пишет в тексте запроса;
  guidance `describe`/handoff, не билдер.
- **Роли полей** — `Период`+`ПорядковыйНомер`, `Измерение` и т.п.: СКД корректно передаёт
  интервалы дат в подзапросы. **[верно]** → ✓ `build_dcs_query_dataset_field(role=...)`
  поддерживает; `describe` должен раскрыть словарь ролей.
- **Не `ВЫБРАТЬ *`** — объявлять только используемые поля. **[верно]** → ✓ наш билдер и так
  требует явный список полей.

## Связи наборов данных

- **Минимизировать связь на стороне 1С** — `ЛЕВОЕ СОЕДИНЕНИЕ` внутри одного запроса быстрее,
  чем связь наборов на сервере приложений. Объединять в один набор, где возможно.
  **[верно]** → ✎/○ guidance: предпочитать один набор; `build_dcs_dataset_link` — крайний
  случай.
- **Параметры связи** — если связь нужна, настраивать «Параметры связи»: СКД делает точечные
  порционные запросы к подчинённому набору, а не грузит его целиком. **[верно, важно]** →
  △ проверить, поддерживает ли текущий `build_dcs_dataset_link` параметры связи; если нет —
  расширить (это и есть перф-смысл связи).

## Ресурсы и вычисляемые поля

- **Функции общих модулей в выражениях — «враг скорости»**: СКД считает их построчно на
  выводе. Переносить математику/склейку/логику в текст запроса. **[верно]** → ✎ guidance;
  △ возможна мягкая эвристика-предупреждение на вызов общих функций в выражении (низкий
  приоритет).
- **`Рассчитывать по…` для итогов** — всегда указывать поле группировки расчёта; итог «по
  всему отчёту» на тяжёлых формулах бьёт по CPU. **[верно]** → △ **пробел билдера**:
  `build_dcs_total_field(data_path, expression)` сейчас не принимает scope «рассчитывать
  по»; добавить, когда встретится reference с ним.

## Вывод для пользователя

- **Представления для наборов «Объект»** (ТаблицаЗначений): для ссылочных колонок передавать
  и текстовое представление (`КонтрагентПредставление`), иначе N+1 запросов на выводе.
  **[верно]** → набор «Объект» (ValueTable-датасет) мы пока не строим (только query-наборы);
  зафиксировать на будущее.
- **Частые параметры/отборы — в быстрые пользовательские настройки**; пользователь должен
  видеть активные фильтры, чтобы не запускать тяжёлый отчёт без отбора. **[верно]** → ✓
  **уже реализовано**: `view_mode='QuickAccess'` + `userSettingID`; дефолт-политика «основные
  отборы в польз. настройки» уже в `vocabulary.guidance`. Эта практика напрямую
  подтверждает недавний срез.

## Итог: что делаем с этим

1. **Guidance «параметры vs отборы» в `describe` — ✓ сделано** (раздел `parameter` в
   `vocabulary.py`, отражает `build_dcs_parameter`). Осталось (○): дефолты `set_dcs`.
2. **Будущий билдер (△):** «рассчитывать по» у `total_field`; параметры связи у
   `dataset_link` (проверить/расширить); набор «Объект» с представлениями.
3. **Авторинг (✎):** `{}`-условия, минимизация связей, математика в запросе — это текст
   запроса от агента; место — guidance/handoff, не билдер.
4. **Не делаем:** запрещающую валидацию по этим эвристикам (перф-советы, а не правила
   формата; зависят от СУБД/версии).
